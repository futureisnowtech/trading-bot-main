Real Learning / ML / RBI Audit

1. Executive truth

The repo contains multiple layers that look like learning:
- trade attribution
- feature snapshots
- retrain queue
- walk-forward models
- online learner
- candidate labeler
- entry priors
- RBI nightly research
- spot edge calibrator

But they are not all equally real in the live spot lane.

For live spot, the current state is:
- candidate-level labeling and priors are real and active
- trade-level spot learning is broken
- ML models are stale
- online learner is dead code
- RBI is only loosely related to the current spot lane
- spot edge calibration is effectively non-operational

2. Loop-by-loop status

2.1 Candidate labeler

Status: REAL and ACTIVE

Evidence
- source: candidate_labeler
- system_events count: 120 INFO rows
- last candidate_labeler event: 2026-04-28T12:10:37.355023-04:00
- candidate_outcomes rows: 19587

What it does
- labels scan_candidates with:
  - ret_15m_pct
  - ret_1h_pct
  - ret_4h_pct
  - MFE / MAE
  - hit_1r / hit_2r / hit_stop
  - timing to 0.5R / 1R / 2R

Why it matters
- this is the most alive learning-like surface for spot right now
- it can still provide valuable pre-trade priors

Key limitation
- it is not learning from actual closed-trade execution truth
- it is labeling forward path from scan candidates, not from trade lifecycle truth

2.2 Entry priors

Status: REAL and ACTIVE, but TARGET-MISMATCHED

Evidence
- learning/entry_priors.py is live-wired into scheduler/v10_runner.py
- it drives the economics gate win-rate estimate

What it learns
- Bayesian win-rate prior from scan_candidates JOIN candidate_outcomes
- current win label:
  - hit_1r = 1
  - and hit_stop = 0

Why this is only partly useful
- the current spot strategy is a fast scalp lane
- many spot exits are thesis-based and shorter-horizon than a clean 1R completion
- entry priors are therefore learning a different target than the live spot lane actually wants

Additional mismatch
- entry_priors buckets by:
  - exchange
  - primary_setup
  - regime
  - direction
- but the spot lane itself uses:
  - spot setup_family
  - spot setup_score
  - spot derivative state

Recent mismatch evidence
- recent entered spot candidates by scanner primary_setup vs spot setup_family:
  - ranging_mr -> pullback_reclaim: 51
  - momentum -> pullback_reclaim: 43
  - blank -> pullback_reclaim: 40
  - kst_cross -> pullback_reclaim: 8

Interpretation
- the prior engine and the spot execution engine are not learning in the same setup language

2.3 Spot trade attribution

Status: BROKEN for live spot

Evidence
- all spot attribution rows: 0
- recent spot attribution rows after 2026-04-22T21:36:39Z: 0
- latest trade_attribution.created_at:
  2026-04-22T21:36:39.390822+00:00

Root cause
- spot_engine.close_spot() logs the SELL and deletes the position
- but it never calls:
  - learning_loop.record_closed_trade()
  - learning.post_trade_analyzer.analyze_closed_trade()

Impact
- no live spot attribution
- no Bayesian spot signal updates
- no spot learning from the recent bad streak
- no new learning truth for the dashboard

2.5 Retrain queue

Status: PARTIALLY REAL, but FRAGMENTED and OPERATIONALLY MISREPORTED

Evidence
- direct queue due count under current schema check: 0 pending >= 20
- raw queue has pending items
- normalized queue shows:
  - SOL LONG = 20
  - ADA LONG = 18
  - ETH LONG = 7
  - DOGE LONG = 8

Fragmentation evidence
- queue uses mixed keys like:
  - SOL
  - PF_ADAUSD
  - PF_ETHUSD
  - PF_SOLUSD

Meaning
- evidence is split across different key styles
- even when underlying exposure is the same, the queue can treat it as separate learning slots

Monitoring blind spot
- nightly_audit.py checks:
  SELECT COUNT(*) FROM ml_retrain_queue WHERE status='pending'
- but ml_retrain_queue has no status column
- nightly audit then silently marks the check pass with:
  "retrain queue check skipped: no such column: status"

Impact
- operational health reporting is falsely reassuring

2.7 ML inference specificity

Status: REAL but GENERIC

Critical code fact
- signal_engine._get_ml_score() calls:
  model_store.predict_ml_score(features, direction)
- it does not pass symbol

Why that matters
- ModelStore only chooses pair-specific models when symbol is provided
- without symbol, it falls back to GENERIC

Impact
- the live ML score is not symbol-aware at inference time
- pair-specific modeling is effectively bypassed

2.10 Spot edge calibrator

Status: EFFECTIVELY NON-OPERATIONAL

Evidence
- spot_edge_conditions rows: 0
- no system_events mentioning spot_edge_calibrator
- latest nightly audit report does not contain a populated spot_edge_calibration section

Even worse: the calibrator SQL is structurally weak
- it reconstructs from SELL trades
- then joins scan_candidates on:
  - exact symbol match
  - decision='entered'
  - timestamp difference < 300 seconds

Why this is broken in practice
- it compares scan candidate time to close time, not to entry time
- exact-symbol matching misses PF_/base-asset translation cases

Empirical damage
- real live spot SELLs matched by the calibrator-style join:
  - total sells: 139
  - matched: 68
  - match rate: 48.9%
- by symbol:
  - SOL: 33 sells, 10 matched
  - XRP: 28 sells, 14 matched
  - ETH: 19 sells, 1 matched
  - LTC: 4 sells, 0 matched

Interpretation
- even if the calibrator were actively run, it would be reconstructing from a badly incomplete sample

3. The nightly audit is not telling the full truth

Examples from persisted audit output:
- learning_health passes because total snapshots are nonzero
- attribution_coverage shows absurd values like 233,787.5%
- ml_retrain_queue passes even though the query is invalid

Interpretation
- some monitoring surfaces are counting legacy or mismatched scopes
- the audit is strong on proof suite and candidate journaling
- it is not yet a reliable source of truth for live spot learning health

4. Bottom line

What is real right now
- candidate labeling
- candidate outcome tables
- entry priors
- stale generic ML inference

What is broken for live spot
- trade attribution
- feature snapshot ingestion
- recent retraining
- online adaptation
- spot edge calibration

What is only nominal
- the idea that recent live spot losses are teaching the bot anything meaningful

At the moment, the live spot bot is not truly learning from its recent bad trades.
